# 影资管家最终高性能版开发文档

文档版本：V2.0 高性能商业版  
文档状态：重构立项版 / 可作为全新项目基线  
目标形态：全中文本地桌面端专业 DIT 素材管理软件  
推荐技术栈：C++23 + Qt 6/QML + FFmpeg 库级集成 + SQLite WAL + CMake  
适用平台：Windows 10/11、macOS 13+，后续可扩展 Linux  
核心目标：高性能导入影视素材目录，自动识别文件结构，生成缩略图，提取视频/音频元数据，导出专业中文 PDF 报表与文件树。

---

## 0. 重要决策

本版本为 **全新高性能重建方案**，不再以 Python/PySide6 原型为基础继续开发。

### 0.1 放弃旧方案的原因

旧方案适合快速验证，但不适合作为长期商业版核心架构：

1. Python UI 与后台任务长期耦合后，性能边界不够清晰。
2. 大量视频缩略图、元数据提取、哈希计算和 PDF 导出对 CPU、I/O、线程调度要求较高。
3. PySide6 可以实现现代界面，但要达到专业 DIT 软件级别的高帧率、大量缩略图虚拟滚动、复杂任务状态和多面板工作区，工程成本会持续上升。
4. 使用 FFmpeg 命令行进程适合验证功能，但正式版需要更细的错误控制、任务取消、批量复用、超时管理和内存控制。
5. 长期商业化需要更稳定的打包、依赖管理、跨平台表现和性能调优空间。

### 0.2 新方案目标

新方案直接面向正式商业版，核心目标：

1. 高性能。
2. 全中文界面。
3. 专业 DIT 视觉工作台。
4. 稳定处理大目录。
5. 可长期维护。
6. 支持插件化扩展。
7. 支持现代 PDF 报表工作室。
8. 对原始素材只读，不破坏用户文件。

# 2. 技术选型总览

## 2.1 最终推荐技术栈

| 层级 | 技术 |
|---|---|
| 主语言 | C++23 |
| UI 框架 | Qt 6 + QML / Qt Quick |
| 桌面平台 | Windows / macOS，后续 Linux |
| 构建系统 | CMake + Ninja |
| 数据库 | SQLite WAL + FTS5 |
| 媒体解析 | FFmpeg libavformat / libavcodec / libavutil |
| 媒体补充解析 | MediaInfo SDK，可选 |
| 缩略图处理 | FFmpeg 解码 + Qt Image / 自研图像管线 |
| PDF 生成 | QPdfWriter + QPainter / 自研 PDF 报表渲染层 |
| PDF 预览 | Qt PDF 模块 / 自研预览适配层 |
| 图表渲染 | QPainter / QML Canvas / 自研轻量图表组件 |
| 任务系统 | 自研任务引擎 + QThreadPool / Qt Concurrent |
| 文件扫描 | C++ std::filesystem + 平台特定优化 |
| 日志 | spdlog 或自研 Qt 日志适配 |
| 测试 | GoogleTest / Catch2 + Qt Test |
| 打包 | CPack / Qt Installer Framework / 平台原生安装包 |

## 2.2 关键技术依据

1. Qt Quick/QML 用于构建现代、流畅、组件化 UI。
2. Qt Concurrent 适合编写多线程任务，避免大量手写底层线程原语。
3. FFmpeg 的 libavformat 负责容器格式解复用，libavcodec 负责音视频/字幕编解码。
4. SQLite WAL 模式适合读写并行场景，读操作不阻塞写操作，写操作也不阻塞读操作。
5. QPdfWriter 可以通过 QPainter 绘制命令生成 PDF，适合构建自研报表渲染器。
6. CMake 是 Qt 6 项目的推荐构建路线之一，适合跨平台 C++ 工程。

8. 数据库写入统一由 Repository 层完成。
9. 媒体处理结果以结构化对象返回。
10. 所有耗时任务都必须可取消、可重试、可记录日志。

## 6.3 文件扫描引擎 ScanEngine

职责：

1. 高性能递归扫描目录。
2. 识别文件和文件夹。
3. 跳过隐藏文件、系统文件和忽略规则。
4. 识别软链接、快捷方式和网络路径。
5. 分批提交扫描结果。
6. 支持暂停、取消、恢复。
7. 记录扫描异常。

扫描策略：

1. 使用 `std::filesystem::recursive_directory_iterator` 作为基础。
2. Windows 可增加长路径处理和 UNC 路径支持。
3. macOS 处理权限提示和外部磁盘访问。
4. 扫描结果先进入内存批次缓冲区。
5. 每批 500–2000 条记录提交给 Repository。
6. UI 每 100–300ms 刷新一次，不按文件逐条刷新。

扫描输出：

```cpp
struct ScanBatch {
    SourceRootId sourceRootId;
    std::vector<FolderRecord> folders;
    std::vector<AssetRecord> files;
    std::vector<ScanIssue> issues;
    ScanProgress progress;
};
```

## 6.5 媒体解析引擎 MediaProbeEngine

职责：

1. 读取视频/音频容器信息。

## 7.3 线程池设计

```text
UI 线程：
只负责界面渲染和交互。

数据库写入线程：
单写入队列，避免 SQLite 写锁混乱。

扫描线程池：
偏 I/O，线程数可略高。

媒体解析线程池：
偏 CPU + I/O，线程数受 CPU 核心和磁盘速度限制。

缩略图线程池：
偏 CPU/GPU/解码，默认并发较低。

哈希线程池：
偏磁盘 I/O，必须限速，避免拖慢用户磁盘。

PDF 导出线程：
独立线程，避免阻塞其他任务。
```

推荐默认并发：

| 任务 | 默认并发 |
|---|---|
| 目录扫描 | 2–4 |
| 元数据提取 | CPU 核心数 / 2 |
| 缩略图生成 | 2–4 |
| 哈希计算 | 1–2 |
| PDF 导出 | 1 |
| 数据库写入 | 1 |

## 7.4 任务调度规则

1. 当前可见素材缩略图优先。
2. 用户手动触发任务优先。
3. 元数据任务优先于报表导出。
4. 网络盘任务降低并发。
5. 大文件哈希任务默认低优先级。
6. 失败任务最多自动重试 2 次。
7. 用户可手动重试失败任务。

# 8. 数据库设计

## 8.1 数据库原则

1. 使用 SQLite 单项目数据库。
2. 每个项目一个 `.cvdb` 文件。
3. 启用 WAL 模式。
4. 启用外键。
5. 写入统一走数据库写入线程。
6. 批量写入。
7. 所有表保留 created_at / updated_at。
8. 原始元数据 JSON 单独保存。
9. Schema 必须版本化。
10. 所有迁移必须可重复验证。

# 14. 性能目标

以下为开发测试目标，不承诺在所有硬件上固定达成，实际结果受磁盘速度、网络盘、视频编码、CPU 核心数和文件数量影响。

## 14.1 测试环境基线

建议基线设备：

```text
CPU：8 核以上
内存：32 GB
硬盘：NVMe SSD
系统：Windows 11 / macOS 14+
素材：本地 SSD
```

## 14.2 性能目标

| 场景 | 目标 |
|---|---|
| 启动空项目 | 3 秒内进入主界面 |
| 打开已有中型项目 | 5 秒内可浏览 |
| 10,000 文件目录扫描 | UI 不冻结，增量显示 |
| 50,000 文件目录扫描 | 可完成，支持取消和恢复 |
| 缩略图生成 | 当前可见素材优先生效 |
| 元数据提取 | 多线程并发，失败不阻塞队列 |
| 技术表格滚动 | 大数据虚拟化滚动不卡顿 |
| PDF 生成 | 大报表自动分页，过程可取消 |
| 数据库查询 | 常用过滤 300ms 内返回首屏 |
| 搜索 | FTS 搜索 500ms 内返回首屏 |

## 14.3 UI 性能原则

1. 素材网格必须虚拟化。
2. 技术表格必须虚拟化。
3. 缩略图按可视区域懒加载。
4. UI 不等待后台任务完成。
5. 大列表不一次性创建全部 QML 对象。
6. UI 状态更新节流，避免高频刷新。
7. 图片缓存使用 LRU 策略。
8. 后台任务事件合并后再推送给 UI。

# 17. 错误处理与日志

## 17.1 错误级别

| 级别 | 中文显示 | 说明 |
|---|---|---|
| Info | 信息 | 普通记录 |
| Warning | 警告 | 可继续，但需注意 |
| Error | 错误 | 单项失败 |
| Critical | 严重错误 | 影响项目或导出 |
| Fatal | 致命错误 | 程序无法继续 |

## 17.2 用户提示原则

错误提示必须包含：

1. 发生了什么。
2. 可能原因。
3. 用户可以怎么做。
4. 是否可以重试。
5. 是否影响其他任务。

示例：

```text
元数据提取失败

文件：B001_C008.mp4
可能原因：文件损坏、编码不受支持或读取超时。
建议：检查文件完整性后重试，或在设置中切换 MediaInfo 解析器。
```

## 17.3 日志格式

```text
[2026-05-30 18:42:11.231] [ERROR] [MediaProbe]
asset_id=123
path=D:/Project_A/Footage/B001_C008.mp4
message_zh=元数据提取失败
detail=FFmpeg timeout after 30000ms
```

## 19.3 FFmpeg 许可注意事项

1. 商业发行应优先使用 LGPL 配置。
2. 避免启用 GPL 或 nonfree 组件，除非明确接受相应许可证影响。
3. 动态链接优先于静态链接。
4. 随安装包附带许可证文本。
5. 提供第三方组件声明。
6. 编码专利问题需要单独评估，不能只看 FFmpeg 许可证。

C++23 + Qt 6/QML + FFmpeg 库级集成 + SQLite WAL + 自研任务引擎 + 自研中文 PDF 报表渲染系统

该方案的核心优势：

1. 性能上限高。
2. UI 现代化能力强。

4. 媒体处理可控。
5. 任务调度清晰。
6. 数据库可靠。

8. 适合长期商业化。

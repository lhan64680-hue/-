# 语义索引多核与非阻塞优化方案

## 目标

- 首次构建或增量更新语义索引时，充分利用 CPU 多核。
- 索引准备阶段继续使用上一代稳定索引，不阻塞搜索。
- 索引最终切换时若发生短暂锁竞争，立即降级为关键词检索。
- 向用户显示文档总数、已完成数量和百分比。

## 处理流水线

1. 从 SQLite 收集文件夹、素材和视觉帧文档。
2. 与已持久化的 `search_document` 比较内容哈希，仅保留变更文档。
3. 不持有进程级索引锁，将文本按批送入 BGE ONNX 模型。
4. ONNX Runtime 使用逻辑处理器数减 2 个线程，为 Qt 界面和系统保留调度空间。
5. 全部向量就绪后，短暂获取索引锁，写入 USearch 和 SQLite 并原子切换索引文件。
6. 主线程中的语义查询只尝试获取锁；锁忙时立即返回降级结果，不等待。
7. 新索引完成后丢弃内存中的旧索引，自动重新执行当前搜索。

## 并发策略

- `processSemanticUpdateMutex` 只用于保证同一进程内最多一个写入流水线。
- `processSemanticIndexMutex` 只保护索引加载、写入和最终切换，不再覆盖分词与 ONNX 推理。
- 搜索通过 `try_lock` 获取索引锁，不得在 Qt 主线程上等待。
- 构建新索引时，已加载的稳定旧索引仍可并发服务查询。

## 性能参数

- `CINEVAULT_BGE_THREADS`：ONNX Runtime 内部运算线程数。默认值为 `QThread::idealThreadCount() - 2`。
- `CINEVAULT_BGE_BATCH_SIZE`：单次 ONNX 批量文档数。默认为 32，允许范围为 1～64。

若设备内存较小，可优先将批量调低为 8 或 16；不建议同时增加 ONNX 线程数和外层并发会话数，否则会发生 CPU 超额调度。

## 真实数据验证

验证设备：Intel Core i7-14700K，20 个物理核心、28 个逻辑处理器。

| 项目 | 结果 |
| --- | --- |
| 素材文档 | 228 |
| 文件夹文档 | 7 |
| 视觉帧文档 | 14,238 |
| 总索引文档 | 14,473 |
| 完整构建时间 | 约 95 秒 |
| 持续 CPU 使用 | 约 25.1～25.4 个等效核心 |
| 构建期间窗口状态 | `Responding=True` |
| 峰值私有内存 | 约 1.9 GB |
| USearch 索引大小 | 31,783,952 字节 |

验证使用安装数据库的只读快照，完成后已删除构建目录中的测试数据库和临时索引，未修改安装目录中的用户数据。

## 回归要求

- 批量向量的维度、归一化和与单条推理的余弦一致性通过自动测试。
- 索引更新停在锁外向量准备阶段时，旧索引查询必须在 2 秒内返回。
- 后台同步必须发布文档总数和进度信号。
- 索引完成后，当前搜索必须自动刷新。

4. 为每项问题给出影响、证据、触发条件、修复方案和验收方法。

- 任意项目数据库位于非专用目录时执行“移入回收站”；
- 反馈服务在公网 HTTP、暴力登录、大附件并发下的行为。

## 4. 总体结论

当前版本功能回归基线是可用的，但不建议在修复 P0 问题前继续开放自动安装或在公网传输真实反馈内容。

风险统计：

| 优先级 | 数量 | 处理要求 |
|---|---:|---|
| P0 / 严重 | 2 | 上线或扩大使用前立即修复；可先关闭对应功能止血 |
| P1 / 高 | 7 | 下一修复周期完成，并补故障注入/并发测试 |
| P2 / 中 | 8 | 纳入随后 1～2 个迭代，建立静态检查门禁 |

最重要的四个结论：

1. 使用反馈固定走公网 HTTP，客户端令牌还出现在 URL 查询参数中，消息、联系方式、项目路径和附件存在明文泄露风险。
2. 自动更新只校验文件名和大小，随后通过 UAC 以管理员权限执行，缺少哈希与发布者签名校验。
3. 重新扫描先删除旧目录和素材，再分批提交新结果；任何中途异常都会留下不完整目录，并级联丢失依附于旧素材的数据。
4. 多个后台任务打开数据库时读取的是可变的“当前项目路径”；项目切换时可能出现“文件缓存写在项目 A、数据库记录写进项目 B”的跨项目污染。

## 5. P0 严重问题

### CV-P0-01 使用反馈链路明文传输，默认管理员密码可直接进入生产配置

**类型：已确认的安全缺陷。**

证据：

- `src/application/FeedbackService.cpp:30` 固定服务地址为 `http://115.231.35.105:3021/`。
- `FeedbackService.cpp:370-374`、`588-591`、`636-640` 把客户端令牌放入查询参数或 WebSocket URL。
- `services/feedback/server/app/main.py:29` 和 `ecosystem.config.cjs:45` 在缺少环境变量时回退到 `change-this-password`。
- `main.py:192-195` 将含令牌的 WebSocket URL返回客户端。
- `main.py:767-772` 配置 `allow_origins=["*"]` 且 `allow_credentials=True`。

影响：

- 昵称、联系方式、应用版本、系统摘要、项目名称、项目绝对路径、反馈文字和附件可被链路中间节点读取或篡改。
- URL 中的令牌可能进入反向代理访问日志、浏览器历史、监控平台和错误报告。
- 生产部署遗漏密码环境变量时，攻击者可使用公开默认密码登录管理员端。
- 反馈会话令牌一旦泄露，可读取、删除或持续监听对应会话。

立即止血：

1. 在 HTTPS/WSS 完成前暂停公网反馈入口，或仅允许受控内网访问。
2. 立即轮换管理员密码、现有客户端令牌，并清理代理/应用日志中的查询令牌。
3. 服务启动时如果密码为空或仍为默认值，直接拒绝启动。

正式修复：

1. 由 Nginx/Caddy 终止 TLS，强制 HTTP 301 到 HTTPS，启用 HSTS；客户端生产配置只接受 HTTPS/WSS。
2. 普通 HTTP API 使用 `Authorization: Bearer`，不再使用查询参数传递令牌。
3. WebSocket 使用一次性、短时效连接票据，或使用 `Secure + HttpOnly + SameSite` Cookie；票据消费后立即失效。
4. 管理员密码改为强制随机密钥或密码哈希，增加登录速率限制、失败锁定和安全审计。
5. 将 CORS 收敛到实际管理站点来源；若桌面客户端不依赖浏览器 CORS，可完全关闭跨域。

验收标准：

- 生产配置中不存在 `http://115.231.35.105`、默认密码和通配 CORS。
- 抓包只能看到 TLS 流量；访问日志、错误日志中没有会话令牌。
- 默认/空密码启动失败；连续错误登录会触发限流。
- 旧令牌全部失效，WebSocket 票据不可重复使用。

### CV-P0-02 自动更新包未验证完整性和发布者，却以管理员权限执行

**类型：已确认的供应链与本地提权风险。**

证据：

- `UpdateService.cpp:187-234` 从 GitHub Release 读取版本、文件名、下载 URL 和大小，没有读取可信哈希或签名。
- `UpdateService.cpp:667-692` 对待安装缓存只检查路径存在和文件名匹配。
- `UpdateService.cpp:962-983` 下载完成只检查文件存在、非空和大小。
- `UpdaterSession.cpp:382-403` 最终通过 PowerShell `Start-Process -Verb RunAs` 提权执行该文件。

影响：

- GitHub 账号/Release 被接管、代理链路异常、更新缓存被同用户进程替换或 QSettings 中路径被篡改时，恶意 EXE 可获得管理员权限。
- 文件大小不能证明内容可信；同长度替换可绕过现有检查。
- 下载完成到提权执行之间存在 TOCTOU（检查后替换）窗口。

立即止血：

1. 在签名校验上线前关闭“自动安装”，保留仅提示并引导用户从可信页面下载。
2. 限制下载 URL 必须是 HTTPS 且属于允许的 GitHub/CDN 主机。

正式修复：

1. 发布独立签名清单，至少包含版本、平台、文件名、字节数和 SHA-256；清单由离线发布密钥签名。
2. Windows 在提权前调用 `WinVerifyTrust` 校验 Authenticode，并固定允许的发布者证书身份/公钥。
3. 只接受规范化后位于应用更新暂存目录内的文件，拒绝 QSettings 指向的任意外部路径。
4. 对暂存目录设置最小权限；每次复用、复制以及提权前都重新计算哈希和校验签名。
5. 缩小检查到执行的时间窗口；可使用已打开的只读句柄或安全复制到受保护目录后执行。

验收标准：

- 安装包任意修改 1 字节后，程序在出现 UAC 前拒绝安装。
- 同名同大小伪造文件、缓存外路径、非允许下载主机均被拒绝。
- 正确签名包可升级，签名过期/撤销策略有明确行为并记录审计日志。

## 6. P1 高风险问题

### CV-P1-03 重新扫描先清空旧数据，中途失败会产生数据丢失和半成品目录

**类型：已确认的数据一致性缺陷。**

证据：`ScanEngine.cpp:139-159` 在遍历前删除该素材源的 `folder_node` 和 `asset_file`；之后 `161` 行起每 500 个文件使用独立事务分批提交。最终失败只把素材源状态标为 `failed`，不会恢复旧数据。

由于素材分析、缩略图或关联记录可能依赖 `asset_file` 并使用级联删除，影响不只是目录列表，还可能包括已经生成的分析结果。

修复方案：

1. 引入 `scan_run_id`/generation 或暂存表。
2. 新扫描全部写入暂存代次，校验汇总成功后在单个事务中切换活动代次。
3. 失败时保留上一完整代次，只清理失败暂存数据。
4. 增加应用崩溃后的暂存代次回收流程。

验收标准：在第 1、499、500、501、最后一批和最终汇总处注入异常，旧素材条数、主键关联、分析数据和可搜索结果均保持不变。

### CV-P1-04 项目切换与后台任务并发时，可能写入错误项目数据库

**类型：高可信潜在缺陷，需要确定性并发测试复现。**

证据：

- `DatabaseManager.cpp:153-161` 的 `openThreadConnection()` 使用成员 `m_databaseFilePath`，即调用当下的“当前项目”。
- `ScanEngine.cpp:106-121` 在线程真正运行后才打开当前数据库。
- `MediaTaskService.cpp:110-112` 捕获了旧项目路径，但 `runMediaJobs()` 在 `218-220` 行仍通过管理器打开当前数据库；缩略图路径却继续使用捕获的旧项目路径。
- `ProjectService.cpp:270-319` 可直接关闭并切换主连接，生产代码没有调用 `ScanEngine::waitForIdle()`。

可能结果：

- 项目 A 的 source ID/asset ID 被写进项目 B。
- 数据库写入项目 B，但缩略图和帧缓存保存到项目 A。
- 项目 A 的迟到信号刷新项目 B 的界面、任务状态或日志。

修复方案：

1. 每个后台任务携带不可变 `ProjectJobContext`：项目 UUID、数据库绝对路径、任务代次、缓存根目录。
2. 提供 `openThreadConnectionForPath(path, uniqueName)`，禁止后台任务读取可变的当前项目成员。
3. 切换项目前进入屏障：取消可取消任务，等待数据库写任务结束，或让旧任务在提交前校验项目代次。
4. 所有完成信号携带项目 UUID，ViewModel 忽略非当前项目事件。

验收标准：人为阻塞 A 的工作线程，切到 B 后再释放；A/B 的数据库、缩略图目录、任务状态、日志均不得交叉。

### CV-P1-05 多处 `QtConcurrent::run([this])` 无任务所有者，退出时可能访问已销毁对象

**类型：高可信潜在崩溃/悬空指针缺陷。**

证据：

- `MediaTaskService.cpp:111-114`、`157-160`；
- `MaterialCatalogSyncService.cpp:970`、`1018`；
- `VideoAnalysisService.cpp:2009`、`2363`；
- `SettingsViewModel.cpp:483`。

这些任务捕获裸 `this`，部分 future 仅创建局部变量后丢弃，类中没有统一取消、等待和析构屏障。应用退出、项目切换或依赖对象先销毁时，工作线程仍可能访问服务、数据库管理器或任务引擎。

修复方案：

1. 建立应用级 `TaskSupervisor`，持有所有 `QFuture`/worker，提供取消令牌、项目代次和关闭屏障。
2. 应用退出顺序固定为：停止接收任务 → 发出取消 → 等待/超时 → 关闭数据库 → 销毁依赖。
3. 回调使用 `QPointer` 只能保护 UI 回调；工作体本身也不能解引用已销毁服务。
4. 对不能安全取消的写事务，必须等待提交或回滚完成。

验收标准：扫描、缩略图、同步、视觉分析和接口测试运行中连续执行退出/切换项目，不崩溃、不死锁、不产生半写入；建议配合 Application Verifier/Page Heap 或 AddressSanitizer 可用环境验证。

### CV-P1-06 项目重命名、移动和删除缺少完整的目录边界与补偿事务

**类型：已确认的破坏性文件操作风险。**

证据：

- `ProjectService.cpp:20-26` 接受目录或任意数据库文件，并以数据库所在目录作为项目根。
- 重命名/移动在目录操作成功后，若目录初始化或项目记录更新失败，直接返回，不把目录移回原位置。
- `ProjectService.cpp:529-568` 删除时把数据库文件所在的整个目录送入回收站。

影响：

- 用户打开位于普通资料目录中的有效 `.cvdb` 后，“删除项目”可能把该目录内所有无关文件一起移入回收站。
- 重命名/移动后半段失败会出现设置仍指向旧路径、项目实际已移动、当前项目未重新打开的分裂状态。

修复方案：

1. 项目根必须有受控 manifest/标记文件，并验证规范路径、项目 UUID、数据库路径和 manifest 一致。
2. 删除前拒绝磁盘根、用户主目录、非专用目录；展示准确目录和无关文件数量，要求二次确认。
3. 将操作实现为带恢复日志的状态机，任何一步失败都执行补偿或在下次启动恢复。
4. 更新数据库、全局索引和设置前先记录 old/new 路径，成功后再清理恢复记录。

验收标准：对非专用目录、根目录、含无关文件目录、只读目录、磁盘满和目标冲突逐一测试；失败后路径、设置、全局索引和当前打开状态必须一致。

### CV-P1-07 反馈附件无需鉴权即可下载，上传实现可被大附件并发拖垮

**类型：已确认的隐私与可用性缺陷。**

证据：

- `services/feedback/server/app/main.py:949-955` 的 `/files/...` 路由没有客户端或管理员鉴权。
- `main.py:733-761` 对每个附件执行 `await upload.read()`，先整体读入内存，再判断是否超过 100MB。
- 附件先写磁盘，再写消息数据库；后续数据库失败或多文件中途超限时，没有统一清理已写文件。

影响：

- 附件 URL 一旦从日志、历史记录或消息转发中泄露，任何人都可下载。
- 多个接近 100MB 的并发附件会产生显著内存峰值，可能导致服务 OOM。
- 请求失败会遗留孤儿文件，长期占满磁盘。

修复方案：

1. 下载路由验证会话令牌/管理员令牌，或使用短时效签名 URL；响应设置 `Cache-Control: private, no-store`。
2. 严格校验 conversation/attachment 为预期 ID 格式，并对 `resolve()` 后路径做 uploads 根目录包含检查。
3. 上传采用 1MB 左右分块流式写入，设置单文件、文件数量、单请求总字节和并发配额。
4. 先写临时文件；数据库提交成功后原子改名，失败统一删除。
5. 对可主动渲染的 HTML/SVG 等类型强制附件下载，并设置 `nosniff`。

验收标准：无权限请求返回 401/403；101MB 单文件、多文件总量超限和数据库故障都不产生残留文件，进程内存保持在设定上限内。

### CV-P1-08 SQLite 工作线程连接没有统一配置，且存在连接仍被引用时移除连接

**类型：已确认的数据库配置缺陷与潜在并发故障。**

证据：

- `DatabaseMigration.cpp:198-202` 只配置 WAL、NORMAL 和 `foreign_keys=ON`，没有 `busy_timeout`。
- `DatabaseManager.cpp:153-161`、`GlobalDatabaseManager.cpp:274-283` 的工作线程连接仅打开数据库，没有调用统一配置；SQLite 外键开关是每连接独立的。
- 多个调用点在局部 `QSqlDatabase db` 仍存活时调用 `removeDatabase`，可能触发 Qt 的“connection is still in use”警告并留下悬空查询。
- 反馈服务 `main.py:125-128` 的 `_connect()` 同样没有为每条连接设置 `foreign_keys=ON`；只有初始化连接在 `135` 行设置，因此会话删除的级联约束并不稳定生效。

影响：外键约束在工作线程中失效、并发写入频繁遇到 `database is locked`、连接名称冲突或关闭时仍有查询引用。

修复方案：

1. 所有连接只能由统一工厂创建，并逐连接设置 `foreign_keys=ON`、`busy_timeout`、WAL 和同步级别。
2. 写冲突增加有上限的指数退避；长任务缩短事务持锁时间。
3. 使用 RAII 连接句柄，确保所有 `QSqlQuery` 和 `QSqlDatabase` 先析构/清空，再调用 `removeDatabase`。
4. 连接名使用 UUID，不依赖可复用的线程 ID。

验收标准：工作线程插入非法外键必须失败；双写者冲突可在超时内恢复；测试期间没有 Qt 连接仍在使用警告。

### CV-P1-09 API Key、反馈令牌和代理凭据明文落盘，视觉接口可默认降级到 HTTP

**类型：已确认的凭据保护缺陷。**

证据：

- `AppSettings.cpp:163-170` 将视觉 API Key 保存到 QSettings。
- `AppSettings.cpp:428-441` 保存含 `client_token` 的反馈会话 JSON。
- 手动代理 URL可包含用户名和密码，同样进入 QSettings。
- `VisionApiClient.cpp:156-180` 对没有 scheme 的地址默认添加 `http://`，之后在 `201-205` 行发送 Bearer API Key。

修复方案：

1. Windows 使用 Credential Manager/DPAPI（或 QtKeychain）保存 API Key、反馈令牌和代理密码。
2. QSettings 只保存凭据引用 ID 和非敏感设置；迁移成功后删除旧明文值。
3. 公网地址强制 HTTPS；仅对 loopback/RFC1918 私网允许用户显式确认的不加密接口。
4. 日志和错误文本统一脱敏 Authorization、URL userinfo 和会话令牌。

验收标准：注册表/配置文件中搜索不到明文凭据；公网 HTTP 视觉地址在发送请求前被拒绝；迁移不丢失现有配置。

## 7. P2 中风险问题

### CV-P2-10 搜索助手存在 A→B→A 竞态，返回 A 后可能永远得不到模型增强

**类型：已确认的异步逻辑缺陷。**

`MaterialCenterViewModel.cpp:1193-1197` 发现同 cache key 正在请求时不再发起新请求；完成回调在 `1227-1231` 先按 generation 判断并丢弃旧结果，缓存写入却在该判断之后。场景为：A 请求未完成 → 输入 B → 又输入 A → 复用旧 A 的 in-flight 标记 → 旧 A 返回时因 generation 不同被丢弃；当前 A 没有新的请求也没有缓存。

修复：结果按不可变 cache key 缓存，与界面 generation 解耦；只有“应用到界面”才校验 generation/current key。或让一个 in-flight 请求维护订阅者列表。增加延迟可控的 A-B-A 单元测试。

### CV-P2-11 UI 线程存在最长数秒的同步等待，启动、搜索和退出可能假死

**类型：已确认的阻塞调用，实际卡顿取决于环境。**

- `LocalSearchAssistantRuntime.cpp:52-60` 在 GPU 探测中同步等待进程启动 3 秒、完成 5 秒。
- 停止模型进程最多同步等待约 2.5 秒（`224-226`）。
- `UpdateService.cpp:367-384` 逐个同步探测代理端口；`570`、`812` 又等待 curl 进程启动 3 秒。
- `MaterialCenterViewModel.cpp:1026-1042` 在 UI 线程同步执行搜索；`993-995` 每次 reload 都重新读取筛选项。

修复：进程和网络探测改为信号驱动/后台 worker；缓存 GPU 和代理结果；搜索使用不可变查询快照在后台执行并以 generation 丢弃过期结果；筛选项只在目录变化时刷新。

验收：在无 GPU 工具、代理黑洞和百万级素材库环境下，主线程事件循环延迟 P95 小于 100ms，取消/退出有明确进度且不冻结窗口。

### CV-P2-12 桌面端附件下载整包进内存，忽略写入失败并提前删除旧文件

**类型：已确认的可靠性缺陷。**

`FeedbackService.cpp:1074-1104` 在网络请求完成后 `reply->readAll()`，没有 Content-Length/实际字节上限；保存前删除目标文件，且忽略 `outputFile.write()` 返回值和关闭错误，磁盘满或部分写入仍回调成功。

修复：在 `readyRead` 中流式写入 `QSaveFile`，限制 Content-Length 与实际累计字节；校验每次写入数量并仅在 `commit()` 成功后替换旧文件；限制 scheme、重定向次数和允许的附件源站。

### CV-P2-13 反馈管理员登录缺少暴力防护，令牌存放在浏览器 localStorage

**类型：已确认的认证加固缺口。**

`main.py:888-894` 直接比较用户名密码，没有限流/锁定；管理员 token 只保存在进程内字典，重启全部失效且过期项没有周期清理；`services/feedback/admin/app.js:195-200` 使用 localStorage，若同源页面发生 XSS，令牌可被读取。

修复：登录限流、失败锁定和恒定时间校验；改用短时效 `HttpOnly + Secure + SameSite` Cookie 并配 CSRF；增加 CSP；过期令牌周期回收。若仍用 Bearer，至少改 sessionStorage 并缩短 TTL。

### CV-P2-14 日志同步逐条打开文件、无轮转、无并发保护

**类型：已确认的性能与磁盘风险。**

`AppBootstrap.cpp:23-37` 的全局 Qt 消息处理器对每条消息同步创建目录、打开并追加 `qml-startup.log`；该处理器实际记录整个运行期而非仅启动期。`Logger.cpp:57-70` 同样逐条打开文件，使用进程级可变路径，没有互斥、大小上限和保留周期。

修复：QLoggingCategory 过滤、单写线程/缓冲队列、按大小和日期轮转、总容量和天数上限、并发保护与敏感信息脱敏。项目切换时让日志上下文随任务固定，避免旧任务写入新项目日志。

### CV-P2-15 QML 静态告警数量过高，已有类型不兼容和布局未定义行为

**类型：已确认的质量门禁缺失。**

7 处 `MiddleDragScrollHandler.flickable`（声明为 Flickable）绑定了静态类型为 `QQuickItem` 的 `ScrollView.contentItem`，涉及 `AssetPreviewOverlay.qml:260`、`InspectorPane.qml:55`、`JobProgressInspectorPane.qml:238`、`SettingsDialog.qml:175` 和 `MaterialCenterWorkspace.qml` 三处。另有 4 处布局元素直接设置 width/height。

这些绑定可能在当前 Qt 实现中运行，但依赖内部实际类型，升级 Qt 或调整控件后可能失效。

修复：给 ScrollView 内部 Flickable 建立受支持且类型明确的引用，或修改组件接口；布局使用 `implicitWidth/implicitHeight` 或 `Layout.preferredWidth/Height`；逐步限定属性访问。CI 先禁止新增告警，再分批把存量降到 0。

### CV-P2-16 代理凭据可能出现在 curl 子进程命令行

**类型：已确认的信息暴露路径。**

`UpdateService.cpp:57-63` 把完整 proxy URL 作为 `curl.exe --proxy` 参数；系统代理用户名和密码在 `299-304` 行写入 URL。其他同用户进程、诊断工具或崩溃采集可能读取命令行。

修复：优先使用 Qt 网络栈或 Windows 安全凭据 API；若必须使用 curl，使用受 ACL 保护且短生命周期的配置/凭据传递机制，并避免 URL userinfo。日志只显示代理类型和主机，不显示用户名/密码。

### CV-P2-17 CMake 项目版本与实际注入版本漂移

**类型：已确认的构建一致性问题。**

`CMakeLists.txt:3` 仍为 `project(CineVault VERSION 0.1.97)`，当前构建缓存却把 `CINEVAULT_APP_VERSION` 注入为 `0.1.164`。全新构建、不同 preset、安装包元数据与运行时版本可能不一致。

修复：建立唯一版本源（如 `version.json` 或顶层 CMake 变量），运行时、安装器、资源、更新资产名和发布脚本全部从该源生成；CI 比对产物版本和 Git tag。

## 8. 可优化方向

### OPT-01 语义索引从“全表收集再增量比对”改为变更日志驱动

`SearchDocumentSyncService.cpp:56-253` 每次同步都会读取全部文件夹、素材、逐帧分析和维度分析，再构造所有文本并与现有 key 比对。数据量增大后 CPU、内存和 SQLite 读负载近似 O(N)。

建议增加 `search_document_change`/dirty key 表或 revision 列；目录同步、分析写入时记录受影响 document key，后台只重建变化项，并周期性执行低频全量校验。

### OPT-02 统一后台任务模型

当前扫描、媒体任务、目录同步、视觉分析和设置测试分别管理 QtConcurrent/QProcess，取消、进度、项目上下文和关闭行为不一致。建议统一成：

`TaskSupervisor -> ProjectJobContext -> Worker -> Transaction/Cache -> Project-scoped event`

统一后可同时解决项目串库、退出崩溃、迟到信号、取消和可观测性问题。

### OPT-03 拆分超大文件，降低回归半径

当前较大的核心文件包括：

- `VideoAnalysisService.cpp`：约 3132 行；
- `MaterialCenterWorkspace.qml`：超过 2000 行；
- `MaterialCenterViewModel.cpp`：接近 2000 行；
- `VisionApiClient.cpp`：约 1563 行；
- `SearchEngine.cpp`：约 1237 行；
- `GlobalDatabaseManager.cpp`：约 1173 行；
- `FeedbackService.cpp`：超过 1000 行。

建议按“编排、持久化、网络、解析、缓存、状态机、视图分区”拆分，并通过小接口注入依赖。先拆 P0/P1 所在文件，不做无行为变化保障的全量重写。

### OPT-04 把 SQLite 访问收敛到仓储/连接工厂

目前服务直接拼接 SQL、管理连接名和事务，造成每连接 PRAGMA 漏配、移除连接时机错误和跨项目路径读取。建议提供：

- `ProjectDatabaseSession(path, mode)`；
- `GlobalDatabaseSession(mode)`；
- `TransactionGuard`；
- 对扫描、项目、素材、分析分别建立 repository。

### OPT-05 建立风险导向的 CI 门禁

建议新增以下流水线：

1. 全新目录 Release 构建，不只增量构建；
2. 33 项现有 CTest + Python/网页测试；
3. `qmllint`，先做“告警不增加”基线；
4. clang-tidy/编译警告门禁并生成 `compile_commands.json`；
5. 扫描故障注入、A/B 项目切换并发、退出压力测试；
6. 更新包篡改/签名测试；
7. 服务端依赖漏洞扫描和 secret scan。

### OPT-06 构建目标与源码清单去重

当前 `CMakeLists.txt` 体积较大，多项测试重复组合源码。建议把核心代码拆成 `cinevault_core`、`cinevault_application`、`cinevault_infrastructure` 等库目标，测试链接目标而不是复制源码列表，可减少漏编译和不同测试间编译配置漂移。

### OPT-07 缓存与发布产物保留策略

输出目录保存了多个约 770MB 级安装包版本。建议保留最近 N 个正式版本和当前回滚版本，其他转存或清理；运行缓存按种类设置配额和 LRU，日志、更新 `.part`、失败分析帧和孤儿附件纳入统一清理器。

## 三、问题根因与修改前后对比

### 原实现

```text
逐文档 tokenize -> 单条 ONNX Run -> 单条向量归一化
搜索、建模和写索引共用一把全局互斥锁
首次搜索在 UI 路径等待后台全量索引任务
CPU 约使用 1 核，窗口触发 AppHangB1
```

### 新实现

```text
批量 tokenize/padding -> [batch, sequence] ONNX Run -> 批量归一化
默认 batch=32，ONNX intra-op 默认 logical CPU - 2
耗时推理阶段不持有搜索锁，旧索引持续服务
最终提交阶段短暂换入新索引
搜索 try-lock 失败立即回退，不等待、不同步重建
```

可调参数：

- `CINEVAULT_BGE_THREADS`：1～256。
- `CINEVAULT_BGE_BATCH_SIZE`：1～64。

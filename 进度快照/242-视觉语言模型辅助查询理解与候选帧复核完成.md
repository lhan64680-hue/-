# 242 - 视觉语言模型辅助查询理解与候选帧复核完成

## 一、本轮目标

在已经完成的本地自然语言、日期、类型、文件夹、素材与独立帧搜索之上，复用现有视觉语言模型接口，实现：

- 模型辅助理解较模糊的自然语言搜索；
- 本地硬条件优先，模型不能篡改“昨天、视频、图片、文件夹、帧”等明确意图；
- 本地结果立即显示，模型后台异步补充；
- 帧搜索对已有 Top-N 候选做真实画面复核；
- 模型不能凭空创造素材 ID 或帧 ID；
- 断网、超时、低置信度和非法输出均自动降级到本地搜索。

## 二、整体执行链

1. `NaturalLanguageQueryParser` 先按系统当前日期解析相对日期、类型、结果域、OCR 和同实体条件。
2. `SearchEngine` 立即执行 SQL、FTS5、BGE/USearch 和严格实体混合检索，首屏不等待网络。
3. `VisionApiClient::understandSearchQuery` 在后台调用现有视觉语言模型，返回受 JSON Schema 约束的查询建议。
4. `SearchQueryUnderstanding::merge` 合并建议，本地明确日期、类型和目标永远优先。
5. 如果结果域是视觉帧，最多把前 8 个已有帧缩略图交给模型复核。
6. 复核响应只能引用候选白名单 ID；虚构和重复 ID 被丢弃，只调整已有帧顺序、置信度和理由。
7. 任一模型阶段失败时保留本地结果或原排序，并在搜索状态栏显示降级原因。

## 三、完成模块与文件

### 模块 1：模型查询理解协议与安全合并

新增：

- `dit-tools-src/cinevault-pro/src/core/search/SearchQueryUnderstanding.h`
- `dit-tools-src/cinevault-pro/src/core/search/SearchQueryUnderstanding.cpp`

修改：

- `src/domain/SearchTypes.h`
- `src/core/search/SearchEngine.h/.cpp`

- `CMakeLists.txt`

协议字段：

- 版本、结果域、语义文本、词法词；
- 素材类型；
- 日期起止、原始日期表达、优先日期字段；
- 文件夹是否由素材条件反查；
- OCR；
- 同一视觉实体的标签、颜色、材质、属性；
- 置信度与简短解释。

校验规则：

- 文本长度、词数、实体数均有上限；
- 结果域、素材类型和日期字段只接受白名单枚举；
- 日期必须为 ISO 日期，起止有效，范围不超过 3660 天；
- 置信度必须在 0～1；低于 0.55 不应用；
- 本地已识别日期、素材类型、文件夹/帧目标不可被模型覆盖；
- 模型实体属性只用于补充，不能删除或弱化本地严格实体条件。

代码前后对比：

- 之前：`SearchEngine` 只执行 `m_queryParser.parse(queryText, referenceDate)`。
- 现在：仍先执行本地解析；只有存在已校验的 `ModelSearchUnderstanding` 时，再调用 `SearchQueryUnderstanding::merge`，随后沿用同一混合检索链。

### 模块 2：异步调用、缓存、超时与降级

修改：

- `src/infrastructure/network/VisionApiClient.h/.cpp`
- `src/ui/viewmodels/MaterialCenterViewModel.h/.cpp`
- `src/ui/qml/workspaces/MaterialCenterWorkspace.qml`
- `src/app/AppContext.cpp`

完成内容：

- 复用设置页已有 Base URL、API Key、模型名和解析超时，不建立第二套模型配置。
- 新协议名：`material_search_understanding_v1`。
- 请求把用户查询当作数据，明确抵抗查询中的提示词注入，并禁止生成路径、项目、素材 ID 和帧 ID。
- 本地结果先显示；模型调用使用 `QtConcurrent` 后台执行，不阻塞界面或数据库主线程。
- 搜索文本每次变化递增查询代次；旧请求晚返回时因代次或原查询不一致而丢弃。
- 查询理解会话缓存最多 64 条，缓存键包含协议、日期、接口、模型和查询。
- 查询理解超时限制在 5～20 秒。
- 界面状态包括：本地规则已完整理解、模型理解中、已辅助理解、缓存命中、未配置、低置信度、失败降级。

代码前后对比：

- 之前：`reload()` 同步调用一次本地 `searchMaterials`。
- 现在：`reload()` 先执行 `executeSearch()` 显示本地结果，再调用 `startSearchUnderstanding()`；模型成功后只在主线程重新执行数据库搜索。

### 模块 3：受约束的 Top-8 候选帧视觉复核

修改：

- `src/domain/SearchTypes.h`
- `src/core/search/SearchQueryUnderstanding.h/.cpp`
- `src/infrastructure/network/VisionApiClient.h/.cpp`
- `src/ui/viewmodels/MaterialCenterViewModel.h/.cpp`

完成内容：

- 新协议名：`material_frame_rerank_v1`。
- 帧结果模式最多发送前 8 个候选缩略图；每张图携带既有 `frameKey` 与有限描述信息。
- 响应包含 `candidate_id`、是否相关、0～1 分数和证据理由。
- `parseFrameRerankPayload` 使用已有帧键白名单过滤响应；未知、虚构、重复、越界分数均不会进入排序。
- 相关候选上移，未复核候选保留在中间，不相关候选下移；不删除本地结果。
- 本地分数与模型复核分数按 70%/30% 融合，置信度按 65%/35% 融合。
- 模型理由以“视觉模型复核”追加到中央帧卡片的命中原因。
- 帧复核会话缓存最多 32 条，缓存键包含模型、查询和候选帧键。
- 复核超时限制在 5～25 秒；失败完整保留原排序。

代码前后对比：

- 之前：帧卡片按本地混合分数排序。
- 现在：本地排序仍作为首屏和失败兜底；模型只能在已有前 8 个候选内提供复核分数，不能返回集合外结果。

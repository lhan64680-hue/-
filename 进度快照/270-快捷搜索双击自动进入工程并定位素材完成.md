# 270 - 快捷搜索双击自动进入工程并定位素材完成

## 本轮目标

把快捷搜索结果双击改成完整的单步导航：自动打开结果所属工程；即使用户只在项目库打开了工程但尚未点击进入，也会自动进入工程；自动切换到素材管理中心；恢复快捷搜索条件；滚动并选中目标素材或目标帧，无需用户继续操作。

## 已完成内容

### 1. 快捷搜索改为单请求导航

- `QuickSearchWindow.qml` 的素材双击仍统一进入 `resultDelegate.activate()`，但普通双击现在只调用一次 `openQuickSearchResultAtIndex(videoKey, index)`。
- QML 不再自行依次执行：
  - `enterProjectFromLibrary()`；
  - 写入 `globalSearchText`；
  - 修改 `currentWorkspace`。
- 文件夹类快捷搜索结果也统一调用 `openQuickSearchFolderResult()`，成功后直接显示主窗口。
- `Ctrl+Enter` 的“在资源管理器中定位文件”语义保持不变。

### 2. Shell 原子进入素材管理中心

- `ShellViewModel` 新增 `enterMaterialCenterFromQuickSearch(searchText)`。
- 方法会在一次调用内完成：
  1. 确认所属工程已打开；
  2. 把 `projectEntered` 设为 `true`；
  3. 恢复全局搜索文字；
  4. 把工作区切换到 `MaterialCenter`；
  5. 最后才向素材管理中心派发搜索。
- 避免旧流程短暂切入素材库、把搜索请求发送到错误 ViewModel 的竞态。
- 如果工程打开失败，则保持在项目库并显示明确错误，不进入空白素材中心。

### 3. ViewModel 与 AppContext 统一协调

- `MaterialCenterViewModel` 新增信号 `quickSearchNavigationRequested(searchText)`。
- 素材所属工程成功打开、项目筛选恢复、目标键设置并完成重载后才发送该信号。
- `AppContext` 将信号直接连接到 `ShellViewModel::enterMaterialCenterFromQuickSearch()`。
- 主窗口显示仍由 QML 处理，工程/页面/搜索/选择状态由 C++ 应用层统一处理。

### 4. 当前工程和跨工程处理

- 如果目标素材属于当前已经打开的工程，跳过重复 `openProject()`：
  - 不清空当前任务；
  - 不重复触发全局目录同步；
  - 直接进入素材管理中心并定位。
- 如果属于另一个工程：
  - 打开目标工程；
  - 抵抗 `projectChanged` 清空全局搜索文字的副作用；
  - 恢复原快捷搜索条件；
  - 限定到目标工程；
  - 重新加载并选择目标。
- 工程路径比较使用绝对清理路径和 Windows 不区分大小写比较，避免同一工程因路径书写差异被重复打开。

### 5. 异步目录同步后的选择稳定性

- 快捷搜索目标键和目标帧号会在 `reload()` 前写入待显示状态。
- 搜索结果应用时优先保留仍可见的快捷搜索目标，而不是使用刷新过程中的临时选择。
- 后台 `MaterialCatalogSyncService` 完成并再次触发目录重载时，目标素材仍保持选中。
- 素材管理中心创建或重新可见时，会根据 `quickSearchRevealRevision`：
  - 定位到目标索引；
  - `positionViewAtIndex(..., Center)` 滚动到可见区域；
  - 显示蓝色高亮定位环；
  - 素材详情选择与列表 `currentIndex` 同步。

## 具体修改前后对比

### QML 双击导航

修改前：

```qml
if (materialCenterViewModel.openQuickSearchResultAtIndex(videoKey, index)) {
    shellViewModel.enterProjectFromLibrary()
    shellViewModel.globalSearchText = searchField.text
    shellViewModel.currentWorkspace = shellViewModel.materialCenterWorkspaceId
    showMainWindow(true)
}
```

修改后：

```qml
if (materialCenterViewModel.openQuickSearchResultAtIndex(videoKey, index)) {
    showMainWindow(true)
}
```

工程进入和页面切换不再由 QML 分步控制。

### Shell 状态切换

修改前：

```text
进入工程 -> 暂时切到素材库 -> 派发搜索 -> 再切素材管理中心 -> 再派发搜索
```

修改后：

```cpp
m_projectEntered = true;
m_globalSearchText = searchText;
m_currentWorkspace = WorkspaceId::MaterialCenter;
emit searchRequested(m_globalSearchText);
```

搜索只在最终工作区确定后派发一次。

### 当前工程重复打开

修改前：

```cpp
m_projectService->openProject(targetDatabasePath, &errorMessage);
```

即使目标工程已经打开也会重新打开，连带清空任务和触发目录同步。

修改后：

```cpp
if (hasOpenProject && sameProjectDatabasePath(currentPath, targetPath)) {
    return true;
}
m_projectService->openProject(targetPath, &errorMessage);
```

### 异步刷新选择保护

修改前：

```text
目录同步触发 reload -> 使用刷新时的当前选择 -> 目标可能被第一条结果替换
```

修改后：

```text
若 quickSearchRevealVideoKey 对应素材/帧仍在结果中
-> 始终优先把它作为 selectedKey
-> 后台同步完成后的第二次 reload 仍保持目标选中
```

## 打包探针诊断说明

- 首次对 staging 强制使用 `QT_QPA_PLATFORM=offscreen` 时出现 `0xC0000409`。
- Windows 事件明确指向 Qt 的 `qFatal`；进一步检查确认 staging 正式包只携带用户运行所需的 `platforms/qwindows.dll`，没有测试专用 `qoffscreen.dll`。
- 从开发 Qt 目录仅提供测试用 `qoffscreen.dll` 后，staging 中的正式主程序、Qt DLL、QML、模型与运行时探针退出码为 0。
- 因此该事件不是 Windows 正常运行缺件，也不是产品崩溃；正式用户启动使用 `qwindows.dll`。
- 本次诊断产生的两个 WER 报告已按报告中的 staging 路径精确验证并删除，未删除用户正式软件的历史报告。
